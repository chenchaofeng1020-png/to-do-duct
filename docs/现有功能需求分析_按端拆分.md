# To-Do Duck 现有功能需求分析文档

## 1. 文档说明

本文基于当前项目中已经落地的产品能力整理，目标是输出一份便于产品讨论、需求复盘和后续迭代的需求分析文档。

本文重点描述：
- 用户是谁
- 解决什么问题
- 主要使用场景
- 功能之间的业务逻辑
- Mac 端与手机端的定位差异

本文不包含：
- 技术方案
- 数据结构设计
- 架构实现方式
- 明显仍处于规划阶段、尚未稳定进入主流程的内容

## 2. 产品定位

To-Do Duck 是一款围绕“今天”和“轻记录”展开的个人效率工具。产品不是想把所有任务管理得非常复杂，而是希望通过更轻量、更柔和的方式，帮助用户把注意力从“长期压力”拉回到“今天要做什么”和“此刻想到什么”。

当前产品由两个核心域组成：
- 待办：按天组织任务，用卡片承载每日事项，强调“今天”的完成感。
- Memo：用于随手记录灵感、想法和临时信息，强调快速输入、快速回看、轻分享。

在此基础上，产品进一步形成了两个非常关键的体验特征：
- 任务可以被“延续”到未来日期，而不是被动过期。
- 不同设备承担不同角色，手机端偏即时记录与轻操作，Mac 端偏持续处理与集中整理。

## 3. 跨端统一需求逻辑

### 3.1 核心用户需求

当前产品主要满足四类需求：

1. 用户希望每天都有一个明确的任务承载空间，而不是被无限增长的列表压住。
2. 用户希望未完成的事情可以自然延续，而不是反复重建或遗忘。
3. 用户希望灵感记录和待办管理相互独立，避免混杂。
4. 用户希望在不同设备上都能延续同一套个人管理习惯，但操作强度因设备而异。

### 3.2 核心业务对象

当前产品的业务核心可以理解为三类内容：

- 每日卡片：承载某一天的待办事项，是待办域的组织单位。
- 待办事项：用户真正要完成的任务，可编辑、完成、删除、延续。
- Memo 卡片：独立的轻记录内容，按时间流方式展示。

### 3.3 统一业务原则

无论在哪个端，当前产品都遵循以下统一原则：

- 待办以“天”为组织单位，不强调复杂项目层级。
- 未完成任务允许延续到后续日期，减少重复录入成本。
- Memo 与 Todo 分离，避免计划型内容和记录型内容互相干扰。
- 强调轻量、即时、低负担，不追求复杂配置。
- 数据默认属于用户个人空间，隐私感和安全感是产品信任基础。

## 4. 手机端需求分析

### 4.1 手机端角色定位

手机端是用户最常打开、最容易发生即时记录行为的入口，因此它承担的是“轻、快、当下”的角色。用户在手机端的核心目标不是深度整理，而是迅速进入、迅速记录、迅速完成和迅速确认。

手机端更像用户日常随身携带的“当天行动面板”。

### 4.2 手机端目标用户场景

手机端主要服务以下场景：

- 早上快速开启今天，查看今天有哪些事要做。
- 白天临时想到一件事，马上补进今天的卡片。
- 某件事今天做不完，希望顺手延到明天或下周一。
- 碎片时间里勾掉一项任务，获得明显完成反馈。
- 突然想到一句话、一件灵感或一个备忘，马上记下来。
- 希望把某条 Memo 以更好看的形式分享出去。
- 不打开 App 也能从桌面小组件快速感知今日进度。

### 4.3 手机端信息架构

手机端当前围绕三个一级入口组织：

- 待办
- Memo
- 我的

这样的结构表达了非常明确的产品逻辑：
- 待办负责“做什么”
- Memo 负责“记什么”
- 我的负责“个性设置和同步偏好”

### 4.4 手机端待办需求分析

#### 4.4.1 新的一天

手机端待办页最重要的起点是“开启新的一天”。

需求本质：
- 用户需要一个非常轻的动作，快速进入今天的计划状态。
- 用户不需要先理解复杂结构，只需要先有一个今天的容器。

当前逻辑特点：
- 首次进入时，产品会自动帮用户创建初始卡片，降低空白页焦虑。
- 后续用户可以主动创建“今天”的新卡片。
- 如果当天已经存在同名日期卡片，系统会自动处理标题区分，避免命名冲突。

对应解决的问题：
- 降低开始规划的门槛。
- 避免“今天从哪里开始记”的决策成本。

#### 4.4.2 每日卡片承载任务

每日卡片是手机端待办的主视觉和主操作单元。

用户需求：
- 希望今天的任务聚焦在一个块里，减少跳转。
- 希望一眼看到当天完成了多少、还剩多少。

当前卡片满足的产品逻辑：
- 卡片顶部承担日期识别与进度概览作用。
- 卡片内部直接支持新增任务，形成“看到卡片就能马上写”的低摩擦体验。
- 卡片本身支持标题编辑与删除，说明卡片不仅是系统生成对象，也允许用户按自己的理解命名与整理。

这意味着卡片在产品中不只是日期容器，还承担“当天主题页”的角色。

#### 4.4.3 任务录入与完成

手机端任务录入强调即时性。

需求场景：
- 用户正在移动、开会、排队、通勤，不适合做复杂操作。
- 用户希望看到输入框就能直接写，写完就进入列表。

当前满足方式：
- 卡片底部直接提供新增入口。
- 输入完成后任务立即进入当前卡片。
- 勾选完成时会触发明显的正向反馈，强化“完成一件事”的心理奖励。

对应的产品价值：
- 把录入动作缩短到最小。
- 把完成动作做成正向激励，而不是机械标记。

#### 4.4.4 任务延续

“延续”是当前产品最有辨识度的核心逻辑之一。

要解决的问题不是“复制任务”，而是：
- 某件事今天没做完，但它并没有失效。
- 用户不希望删除旧任务重建新任务。
- 用户希望知道一件事已经连续跟进了多少天。

当前需求逻辑包括：
- 用户可从任务操作入口发起延续。
- 系统默认引导到更合理的未来日期，而不是要求用户重新建一条任务。
- 选择日期时提供快捷选项和日历选择，兼顾效率与灵活性。
- 延续后的任务会形成一条连续关系，并通过连续天数标记强化“这件事还在推进中”的认知。
- 默认不鼓励往过去日期延续，避免时间逻辑混乱；只有在设置中主动放开后，才允许这种更高阶的操作。

对用户的真实意义：
- 减少重复录入。
- 让“没做完”变成“继续推进”，降低挫败感。
- 让长期推进中的事项具备轻度连续感，而不必进入重型项目管理。

#### 4.4.5 任务编辑与优先级判断

手机端在任务操作上采取“轻主流程 + 集中动作面板”的方式。

这背后的需求判断是：
- 手机屏幕空间有限，不适合把所有动作长期暴露在列表上。
- 但用户又确实需要调整任务重要性、编辑内容、删除任务、继续延续。

因此当前产品将这些动作收纳进任务操作面板中，包括：
- 编辑任务
- 删除任务
- 设置优先级
- 延续到未来日期

其中优先级的价值不在于做复杂排序，而在于帮助用户在当天任务里快速区分轻重缓急。

#### 4.4.6 卡片内排序

手机端支持同一卡片内任务拖动排序。

这项能力服务的不是“精细管理”，而是非常朴素的需求：
- 用户希望把最重要或最先要做的事放到前面。
- 用户希望按自己的思维顺序组织当天事项。

因此排序更多是“日内组织感”能力，而不是项目管理式排程能力。

### 4.5 手机端 Memo 需求分析

#### 4.5.1 Memo 的存在价值

Memo 与 Todo 分离，说明产品明确区分了两种信息：

- Todo 是有完成预期的行动项。
- Memo 是暂时不需要立刻执行、但需要被记住的内容。

这解决的是很多轻量效率产品里的常见问题：
- 用户一旦把所有内容都记在待办里，列表就会被想法、备忘、提醒和行动混在一起。

因此 Memo 页的产品价值是：
- 给用户一个不带执行压力的记录空间。

#### 4.5.2 Memo 录入场景

手机端 Memo 主要承接以下瞬时场景：

- 临时灵感
- 会议里一句值得记下的话
- 某个待会儿可能会用到的信息
- 一条想法，暂时不想变成任务

当前产品通过悬浮按钮进入 Memo 输入页，这说明设计重点是：
- 平时先看已有内容
- 有需要时快速拉起输入

这样的结构更适合移动端碎片场景。

#### 4.5.3 Memo 浏览与回看

当前 Memo 以卡片流方式展示，按时间回看。

需求逻辑是：
- 用户记录 Memo 时通常不需要复杂分类，只希望能快速翻回最近内容。
- 卡片化形式比纯文本列表更容易形成“片段记忆”，也更适合承载分享。

因此 Memo 页当前更像轻量信息流，而不是重型知识库。

#### 4.5.4 Memo 编辑、复制、分享、删除

手机端 Memo 的后续动作已经比较完整，说明这部分不只是“记一下”，而是逐步具备内容再利用能力。

当前已满足的需求包括：
- 编辑：记录后仍可修订，适合整理表达。
- 复制：便于二次粘贴到聊天、文档或其他应用。
- 删除：便于清理无效内容。
- 分享：支持将 Memo 以视觉化卡片方式导出和分享。

其中“分享”是一个很重要的产品信号：
- Memo 不只是个人私用，也考虑用户希望把一句话、一段想法进行外发传播的场景。
- 分享样式的存在，使 Memo 从“记录工具”向“表达工具”延伸了一步。

### 4.6 手机端“我的”与设置需求分析

手机端设置目前聚焦在少数但关键的控制项上。

这说明当前产品策略不是让用户做大量配置，而是只开放直接影响使用体验的开关。

当前核心设置需求包括：
- 是否允许任务延续到过去日期
- 是否开启云同步
- 版本信息查看

对应的产品意义：
- “是否允许过去日期延续”服务于少数有补记、补规划习惯的用户。
- “云同步”服务于跨设备连续使用的用户。
- 设置项少，能降低理解和维护成本。

### 4.7 手机端桌面小组件需求分析

虽然小组件不属于 App 内页面，但它是手机端体验的重要延伸。

小组件承担的需求不是编辑，而是“看一眼就知道今天什么状态”。

当前已承接的场景包括：
- 不进 App 先确认今天任务进度
- 快速感知今日是否还有未完成事项
- 在首页直接看到前几条任务
- 从小组件跳回 App 主页面

其核心价值有两点：
- 降低打开 App 的必要性，提高陪伴感。
- 让任务管理从“主动打开才可见”变成“自然出现在视野里”。

## 5. Mac 端需求分析

### 5.1 Mac 端角色定位

Mac 端不是手机端的简单放大版，而是更偏“持续处理”和“集中整理”的工作台。

当用户坐到电脑前时，通常意味着：
- 有更完整的注意力
- 要处理更多条目
- 更适合做整理、排序、批量移动和周期安排

因此 Mac 端承担的是“桌面效率中枢”的角色。

### 5.2 Mac 端目标用户场景

Mac 端主要服务以下场景：

- 开始一天工作时，集中安排今天的任务。
- 将脑子里还没确定日期的事情先丢进收集箱。
- 在不同日期卡片之间拖动任务，重新调整安排。
- 设置某项任务在未来一段时间内重复出现。
- 一边查看历史卡片，一边持续补充和整理。
- 在更大的屏幕上持续记录和搜索 Memo。

### 5.3 Mac 端整体结构

Mac 端当前采用侧边导航，分为：
- 待办
- Memo
- 我的

在待办主界面中，又形成了“双区结构”：
- 左侧是按天展开的任务卡片区
- 右侧是收集箱

这个结构非常符合桌面端认知方式，因为它把“已经安排好的事情”和“尚未安排的事情”明确区分开了。

### 5.4 Mac 端待办需求分析

#### 5.4.1 今日计划与连续卡片

Mac 端同样以每日卡片为核心，但与手机端相比，更强调持续浏览和集中处理。

用户需求：
- 在一个大视野里看到多个日期卡片。
- 在处理今天事项时，也能兼顾前后几天的安排。

因此 Mac 端更适合承担“排布”和“整理”任务，而不仅是快速打勾。

#### 5.4.2 收集箱

收集箱是 Mac 端最重要的差异化能力之一。

它解决的是一个非常常见但在手机端不一定需要强处理的问题：
- 用户经常知道“有这件事”，但当下并不想决定它是哪一天做。

如果所有这类内容都直接进入今天卡片，会造成当天过载；如果不记，又容易丢失。

因此收集箱的需求价值非常明确：
- 先收住，再安排。

当前收集箱承担的逻辑包括：
- 暂存未定日期的待办
- 支持直接新增
- 支持完成、编辑、删除
- 支持与日期卡片之间来回拖动
- 支持在收集箱内部重新排序

这意味着收集箱不是简单的“垃圾桶”或“草稿区”，而是一个真正的“待分配池”。

#### 5.4.3 拖拽式整理

Mac 端支持通过拖拽在不同区域间移动任务，这是桌面端产品价值的重要体现。

它满足的不是炫技式交互，而是高频真实需求：
- 把某个任务从一个日期挪到另一个日期
- 把某个任务先挪回收集箱，稍后再安排
- 在同一张卡片里重新排顺序
- 在收集箱里调整临时事项顺序

桌面端使用鼠标和大屏幕，天然更适合这类“空间整理型”操作。因此这部分能力应被视为 Mac 端效率提升的核心，而不是附属功能。

#### 5.4.4 任务延续

Mac 端同样支持任务延续，但它的使用语境与手机端略有不同。

在 Mac 上，延续更多发生在以下场景：
- 用户复盘今天工作后，主动把未完成事项延到明天
- 用户提前把任务延到某个未来日期
- 用户在处理长期推进事项时，希望保留连续性

因此 Mac 端的“延续”偏向整理场景，手机端的“延续”偏向即时处理场景。

#### 5.4.5 重复任务

重复任务目前是 Mac 端的重要增强能力。

它满足的核心需求是：
- 有些事项并不是“今天做一次就结束”，而是在一个时间范围内需要持续出现。

典型场景包括：
- 每天都要做的习惯项
- 一段时间内持续跟进的工作项
- 临时阶段性任务，例如某个周期内每天检查或打卡

当前需求逻辑体现为：
- 用户从现有任务出发设置重复，而不是单独进入复杂规则中心。
- 用户更关注“重复到什么时候”，而不是配置过多规则细节。
- 对已设置重复的任务，系统会给出明显标识，帮助用户理解它的来源。

这说明当前重复任务设计的重点是：
- 尽量轻量
- 足够解决阶段性连续出现的问题
- 避免一开始就把产品推向复杂日程系统

#### 5.4.6 快捷操作露出

与手机端相比，Mac 端更适合在悬停状态下暴露更多快捷动作。

其需求逻辑是：
- 桌面端用户操作精度更高，能接受更丰富的局部工具条。
- 用户会频繁编辑、删除、移动、延续、设置重复，因此需要更短路径。

当前这种设计兼顾了两点：
- 平时界面保持整洁
- 需要处理时又能快速出现操作按钮

### 5.5 Mac 端 Memo 需求分析

#### 5.5.1 Mac 上的 Memo 角色

Mac 端 Memo 更偏“持续输入和集中回看”。

与手机端相比，Mac 端用户更可能：
- 连续记录多条内容
- 写得更长
- 一边搜索一边翻看历史

因此 Mac 端 Memo 更接近轻量写作面板。

#### 5.5.2 快速连续记录

当前 Mac 端 Memo 页把输入区直接固定在主界面中，而不是每次弹出单独录入页。

这背后的需求判断很明确：
- 桌面端适合连续输入
- 用户可能短时间内记录多条信息
- 输入完成后仍希望光标留在输入区域，保持记录节奏

这使 Mac 端 Memo 更适合承担“持续捕捉想法”的角色。

#### 5.5.3 搜索与回溯

Mac 端 Memo 支持搜索，这说明桌面端不仅承担记录，也承担回查。

对应需求场景：
- 用户记得写过某句话，但不记得具体时间
- 用户想快速翻出某类内容继续修改或复制

因此搜索能力在 Mac 端是高价值基础能力，而不是锦上添花。

#### 5.5.4 文本再利用

Mac 端 Memo 当前支持编辑、复制、删除，且支持文本选中。

这说明 Mac 端 Memo 的产品角色不只是“存着”，而是服务于内容流转：
- 复制到聊天
- 粘贴进文档
- 继续加工

桌面端更强调“写出来后还能马上拿去用”。

### 5.6 Mac 端“我的”与设置需求分析

Mac 端设置延续了手机端的关键控制项，但在产品感觉上更偏“桌面工作环境配置”。

当前承接的核心需求包括：
- 开关云同步
- 控制是否允许任务往过去日期延续
- 查看账号和版本相关信息

这类设置虽然不多，但对于桌面端来说意义较大，因为用户更可能在 Mac 上进行“环境层面”的确认和调整。

## 6. Mac 端与手机端的定位差异

### 6.1 手机端更偏即时性

手机端关注的是：
- 今天要做什么
- 现在顺手记一下
- 此刻勾掉一项
- 快速把没做完的事延后

它适合碎片化、高频、低负担的操作。

### 6.2 Mac 端更偏整理性

Mac 端关注的是：
- 把任务分配到合适位置
- 处理未定日期事项
- 调整顺序和归属
- 设置阶段性重复任务
- 长时间连续记录 Memo

它适合深度处理和结构化整理。

### 6.3 两端协同关系

从现有能力看，产品已经形成比较清晰的协同关系：

- 手机端负责随时进入和轻操作
- Mac 端负责集中整理和长时间使用
- 小组件负责提醒与陪伴

这意味着未来产品讨论不应把两端做成完全相同的功能镜像，而应继续强化各自最自然的使用场景。

## 7. 当前版本的产品价值总结

基于现有功能，To-Do Duck 当前已经形成了比较明确的产品主张：

- 用“每日卡片”降低待办焦虑
- 用“任务延续”替代过期挫败感
- 用“Memo 分区”承接不适合放进待办的内容
- 用“手机轻操作 + Mac 深整理”形成自然的跨端分工
- 用“小组件、反馈动效、分享样式”增强陪伴感和完成感

从需求分析角度看，当前产品最值得继续坚持的方向不是功能越做越重，而是把以下三件事持续做深：
- 对“今天”的聚焦
- 对“未完成但仍在推进”的包容
- 对“随手记录与随手整理”的低门槛支持

## 8. 附：当前已识别的功能范围边界

为保证本文聚焦“现有功能”，以下内容暂不作为当前正式需求范围：

- 复杂项目管理
- 团队协作
- 多层级标签体系
- 完整的日程日历系统
- 明确进入主流程的高级统计分析
- 明确在当前主界面稳定可用的 Memo 置顶、归档等深层管理能力

也就是说，当前产品更适合被定义为：

一款以每日卡片为核心、兼顾轻量 Memo 和记忆延续感的个人任务与记录工具。
